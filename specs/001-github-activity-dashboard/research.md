# Research: GitHub活動実績ローカルダッシュボード

## 決定1: 認証・取得経路は gh CLI + GraphQL を採用

- 決定: Node.js から gh CLI を呼び出し、GraphQL APIで必要データを取得する。
- 根拠:
  - 制約として「PATではなくGitHub CLI利用」が明示されている。
  - Enterprise Cloud / 個人アカウントの切替を gh auth context で扱える。
  - GraphQLは必要項目のみ取得しやすく、集計用途に適する。
- 検討した代替案:
  - Octokit + PAT: 制約違反のため不採用。
  - REST APIのみ: 多段取得が増えやすく、横断集計の効率で不利。

## 決定2: 配備方式は GitHub Pages の静的フロント + ローカル収集CLI

- 決定: 表示系は Viteビルド成果物を GitHub Pages に配備し、データ収集はローカルCLIで実行する。
- 根拠:
  - Pagesは静的ホスティング前提であり、常駐APIサーバを持たない構成と相性が良い。
  - Local First要件を満たしやすく、保存JSONの再利用が容易。
- 検討した代替案:
  - 単一Nodeサーバで収集/表示を兼務: 運用が重く、個人利用で過剰。
  - Electron化: 将来選択肢として有効だが、初期フェーズでは複雑。

## 決定2.1: 収集実行はローカルシェルスクリプトを採用

- 決定: GitHubアクセスとJSON出力は `scripts/export-activity-json.sh` を実行して行う。
- 根拠:
  - 利用者操作を単純化し、実行手順を固定化できる。
  - APIサーバを持たない構成と整合する。
- 検討した代替案:
  - ローカルHTTP API経由: HTTP契約定義は可能だが、現要件では過剰。

## 決定3: ローカル保存形式はバージョン付きJSON

- 決定: datasetVersion を持つJSONを永続化し、互換性検証を可能にする。
- 根拠:
  - 再集計・再表示の再現性確保に必要。
  - 将来のフィールド追加時に前方/後方互換の判定ができる。
- 検討した代替案:
  - CSV: 階層データ（PR-issue参照、レビュー詳細）の表現が難しい。
  - SQLite: 個人用途としては導入・配布コストが高い。

## 決定4: フロントエンド技術は Vue3 + Vuetify + TypeScript + MDI

- 決定: ダッシュボードUIは Vue3 と Vuetify を採用し、アイコンは MDI を利用する。
- 根拠:
  - 指定技術スタックに一致。
  - 指標カード・テーブル・フォーム等のUI実装を短時間で整備できる。
- 検討した代替案:
  - 素のVue + CSS: 実装自由度は高いが、UI整備工数が増える。
  - React系: 指定スタック外。

## 決定5: テスト戦略はコア計算ユニット最優先

- 決定: 集計・派生指標計算を pure 関数化し、Vitestでユニットテストを最優先とする。
- 根拠:
  - 憲章のテスト容易性原則に一致。
  - データ正確性がプロダクト価値の中心。
- 検討した代替案:
  - E2E中心: 回帰検知が遅く、失敗切り分けが難しい。

## 決定6: データ仕様は JSON schema を採用

- 決定: 入出力仕様は `dataset-schema.json` で管理し、スキーマ検証テストで担保する。
- 根拠:
  - 実行モデルが「スクリプトでJSONファイルを出力→UIで読込」のため、HTTP契約定義を不要化できる。
  - Local Firstに必要な互換性管理に直結する。
