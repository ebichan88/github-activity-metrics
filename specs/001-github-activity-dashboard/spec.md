## 1. 入力

ユーザー説明（要約）:

- GitHub活動実績を集計し、MBOや評価資料作成に活用できるローカル向けダッシュボードを作成したい。
- 対象は GitHub Enterprise Cloud と個人GitHubアカウントの両方。
- 複数リポジトリ横断で、期間・担当者単位の集計を行う。
- 認証はPATではなくGitHub CLIを利用する。
- ログイン機能はスコープ外。

## 2. 目的Why

GitHub上の活動実績を一定期間で定量化し、MBOや評価資料にそのまま転用できる形で確認可能にする。
特に、複数リポジトリを横断した担当者別の実績比較を、短い手順で実行できることを目的とする。

## 3. 要求Must / Should / Could

### 3.1 Must

- 指定期間・担当者・複数リポジトリを条件として、PR、Commit、Review、コード変更量を集計できること。
- PR関連で、作成PR数、Closed数、Merged数、PR本文から抽出したIssue参照一覧を表示できること。
- Commit数、additions、deletions を担当者単位で表示できること。
- レビューしたPR数、レビューコメント数を担当者単位で表示できること。
- 派生指標（レビュー率、平均PRサイズ、平均レビューコメント数、マージ率）を算出できること。
- 認証はGitHub CLIを利用し、PAT入力を必須にしないこと。
- 取得済みデータをローカルに保存し、保存データのみで再集計・可視化できること。

### 3.2 Should

- GitHub Enterprise Cloud と個人GitHubアカウントを同一フローで切り替えて集計できること。
- 画面上の操作を最小化し、データ読み込み後すぐに主要指標が確認できること。
- 入力エラーや取得失敗時に、ユーザーが次に取るべき行動を明確な文言で提示すること。

### 3.3 Could

- 評価資料向けに、集計結果のサマリーを期間比較で表示できること。
- 指標ごとの並び替えや絞り込みをダッシュボード上で行えること。

## 4. ユーザーストーリー

### 4.1 US1: 期間指定で活動実績を集計する [P1]

評価資料を作成する担当者として、対象期間とリポジトリ群を指定して、担当者別の活動実績を一度に確認したい。

優先理由: 主要価値である「評価に使える定量集計」を最短で実現するため。

独立テスト: 指定期間と複数リポジトリを入力し、担当者別の必須指標がすべて表示されることを確認できれば完了。

### 4.2 US2: ローカル保存データで再表示する [P2]

同じ担当者として、再集計時にGitHubへ再接続せず、保存済みデータだけでダッシュボードを再表示したい。

優先理由: Local First要件により、再現性と運用コスト削減に直結するため。

独立テスト: オフライン状態でも保存済みデータ読み込みにより前回同等の集計結果が表示されることを確認できれば完了。

### 4.3 US3: 指標を評価資料向けに確認する [P3]

評価資料作成者として、派生指標を含む結果を見て、担当者の活動傾向を短時間で把握したい。

優先理由: 集計結果を意思決定に使うための解釈支援として重要だが、集計基盤より後で提供可能なため。

独立テスト: 派生指標の算出結果と元データの整合が確認でき、評価観点に沿って閲覧できれば完了。

## 5. 受け入れ条件AC

### 5.1 US1の受け入れ条件

1. Given 集計対象期間、担当者、複数リポジトリが指定されている
   When 集計を実行する
   Then 担当者ごとにPR数、Closed数、Merged数、Issue参照一覧、Commit数、additions、deletions、レビュー数、レビューコメント数が表示される

2. Given PR本文に Fixes #123 や Closes #456 が含まれている
   When PR関連情報を表示する
   Then Issue参照一覧としてPRごとに抽出・表示される

3. Given ある担当者の作成PR数が0件である
   When 派生指標を表示する
   Then ゼロ除算エラーを起こさず、レビュー率とマージ率は算出不可として扱う

### 5.2 US2の受け入れ条件

1. Given 事前にローカル保存された集計用データが存在する
   When 保存データを読み込む
   Then GitHubへの通信なしで同一形式のダッシュボードが表示される

2. Given 保存データの形式が不正である
   When 読み込みを実行する
   Then 読み込み失敗理由と修正のヒントが表示され、画面は異常終了しない

### 5.3 US3の受け入れ条件

1. Given 集計済みの基本指標がある
   When 派生指標を表示する
   Then レビュー率、平均PRサイズ、平均レビューコメント数、マージ率が定義式どおり計算される

2. Given 複数担当者の結果がある
   When ダッシュボードを確認する
   Then 各担当者の比較が可能な一覧で確認できる

## 6. エッジケース/失敗時の挙動

- GitHub CLI が未ログインまたは無効セッションの場合:
  - 集計を開始せず、ログイン再実行手順を表示する。
- 指定期間に対象データが存在しない場合:
  - 0件として正常表示し、エラー扱いにしない。
- API制限または一時的な取得失敗が発生した場合:
  - 取得失敗範囲を明示し、再試行可能な状態を維持する。
- リポジトリの一部にアクセス権がない場合:
  - アクセス不可リポジトリを一覧表示し、取得可能分のみで集計を継続する。
- 保存JSONが古い仕様の場合:
  - 読み込み時に互換性不足を通知し、再取得が必要であることを明示する。

## 7. 用語

- 活動実績: PR、Commit、Review、変更量および派生指標の総称。
- 担当者: 集計対象となるGitHubユーザー。
- レビュー率: レビュー数 / 作成PR数。
- 平均PRサイズ: (additions + deletions) / PR数。
- 平均レビューコメント数: レビューコメント数 / レビュー数。
- マージ率: Merged PR数 / 作成PR数。
- ローカル保存データ: GitHubから取得してローカルPCに保存された集計用JSONデータ。

## 8. 機能要件

### 8.1 Functional Requirements

- **FR-001**: システムは対象期間、担当者、複数リポジトリを指定して活動実績を集計できなければならない。
- **FR-002**: システムはPR関連として作成PR数、Closed数、Merged数を担当者単位で集計しなければならない。
- **FR-003**: システムはPR本文から Fixes/Closes 等のIssue参照を抽出し、PRに紐づく一覧として表示できなければならない。
- **FR-004**: システムはCommit数、additions、deletionsを担当者単位で集計しなければならない。
- **FR-005**: システムはレビューしたPR数とレビューコメント数を担当者単位で集計しなければならない。
- **FR-006**: システムはレビュー率、平均PRサイズ、平均レビューコメント数、マージ率を定義式どおり算出しなければならない。
- **FR-007**: システムは認証手段としてGitHub CLIを利用し、PAT直接入力を必須要件にしてはならない。
- **FR-008**: システムは取得済みデータをローカル保存し、保存データのみで再集計・可視化を実行できなければならない。
- **FR-009**: システムはデータ欠損、認証失敗、アクセス権不足などの失敗時に、ユーザー向けの回復可能なエラーメッセージを表示しなければならない。
- **FR-010**: システムはログイン機能を提供してはならない（Out of Scope）。

### 8.2 Constitution Alignment

- 過度な抽象化を避け、個人利用に適した最小構成を採用する。
- 可読性重視で責務を分離し、同一集計ロジックの重複実装を避ける。
- コア集計ロジック（各種件数集計、派生指標算出）に対してユニットテストを用意する。
- 外部通信処理をモック可能な境界で分離し、オフライン検証可能にする。
- Local First要件として、保存済みデータのみの再集計経路を必須とする。
- 主要操作は少ない手順で完了し、複雑な設定を初期利用に要求しない。

### 8.3 Key Entities

- **集計ジョブ**: 集計期間、対象リポジトリ、対象担当者、実行時刻を持つ1回の集計単位。
- **担当者集計結果**: 担当者単位の基本指標と派生指標を持つ結果データ。
- **PRサマリー**: PR状態、関連Issue参照、変更量を持つPR単位データ。
- **レビューサマリー**: レビュー件数、コメント件数、対象PR情報を持つレビュー単位データ。
- **ローカルデータセット**: ダッシュボード描画に必要な保存済みJSON一式。

## 9. 成功基準

- **SC-001**: 初回利用者が対象期間とリポジトリ群を指定して、5分以内に担当者別の主要指標を表示できる。
- **SC-002**: 正常データがある集計実行の95%以上で、必須指標（PR/Commit/Review/変更量/派生指標）が欠損なく表示される。
- **SC-003**: 保存済みデータを使う再表示において、ネットワーク未接続時でも100%のケースで表示が完了する。
- **SC-004**: 利用者の自己評価で、評価資料作成準備に要する集計時間が従来比50%以上短縮される。

## 10. 前提条件

- 利用者は対象組織および対象リポジトリに対して読み取り権限を持つ。
- GitHub CLIは利用者環境で導入済みで、必要な認証状態を確立できる。
- 本機能はローカル利用を前提とし、ログイン機能やマルチユーザー管理は対象外とする。
- 保存データ形式は将来の再集計利用を想定し、最低限の互換性維持方針を持つ。
