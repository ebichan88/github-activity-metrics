## 📥 1. 入力

ユーザー入力:

- 不要ファイルの削除対象として scripts/measure-report-prep-time.sh を指定。
- 担当者別実績の欄で、作成したPR・マージしたPR・レビューしたPRについて、具体的なPR番号を確認できるUI（アコーディオンまたはモーダル）を追加したい。
- GitHub Projectで利用しているissueを収集し、特定期間における担当者ごとのDoneチケット数とEstimate合計を取得したい。
- 既存のPRベースダッシュボードに加え、PRベース用とissue用でメニューを分けたい。

## 👤 2. ユーザーストーリー

### 2.1 US1: PRベース実績の詳細確認 [P1]

評価資料作成者として、担当者別実績の各指標から対象PR番号をすぐに確認し、集計結果の根拠を短時間で追跡したい。

優先理由: 既存ダッシュボードの信頼性を高める中核機能であり、日次運用と説明責任に直結するため。

独立テスト: 任意の担当者で「作成PR」「マージPR」「レビューPR」の各表示からPR番号一覧を開き、番号から対応PRを識別できれば完了。

### 2.2 US2: Issueベース実績の可視化 [P2]

プロジェクトマネージャーとして、GitHub Projectのissueを期間指定で集計し、担当者ごとのDone件数とEstimate合計を確認したい。

優先理由: PRだけでは把握しづらいタスク消化実績を可視化でき、評価観点の網羅性が上がるため。

独立テスト: 期間と対象Projectを指定して収集したデータで、担当者別のDone件数とEstimate合計が表示されれば完了。

### 2.3 US3: 運用整理と不要資産の削減 [P3]

運用担当者として、使わない計測スクリプトを削除し、ダッシュボード入口をPR用とIssue用に整理して操作迷いを減らしたい。

優先理由: 機能価値への直接影響はP1/P2より低いが、運用性と保守性を継続的に改善するために必要。

独立テスト: 不要スクリプトがリポジトリから削除され、UI上でPR用・Issue用メニューを選択して各画面へ遷移できれば完了。

## ✅ 3. 受け入れ条件

### 3.1 US1の受け入れ条件

1. Given 担当者別実績に作成PR数が表示されている
   When 作成PRの詳細表示を開く
   Then 期間条件に一致するPR番号一覧を確認できる

2. Given 担当者別実績にマージPR数が表示されている
   When マージPRの詳細表示を開く
   Then マージ済みPR番号のみが一覧で表示される

3. Given 担当者別実績にレビューPR数が表示されている
   When レビューPRの詳細表示を開く
   Then レビュー実施対象のPR番号一覧を確認できる

### 3.2 US2の受け入れ条件

1. Given GitHub Project issueデータが収集済みである
   When 期間を指定してissueダッシュボードを表示する
   Then 担当者ごとのDone件数が表示される

2. Given 各issueにEstimateが設定されている
   When issueダッシュボードを表示する
   Then 担当者ごとのEstimate合計が表示される

3. Given PRダッシュボードとissueダッシュボードが提供される
   When 利用者がメニューを切り替える
   Then それぞれの集計ビューへ迷わず遷移できる

4. Given GitHub Pages上でダッシュボードを開いている
   When `JSON読込` ボタンからローカルで生成したJSONファイルを選択する
   Then PRビューとIssueビューの表示が開始される

### 3.3 US3の受け入れ条件

1. Given リポジトリに不要スクリプトが残っている
   When 機能変更を適用する
   Then scripts/measure-report-prep-time.sh が削除される

2. Given 既存の利用者導線がある
   When 変更後のUIを開く
   Then PR用とIssue用の入口が明確に分離されている

## 📝 4. 要求

### 4.1 Must / Should / Could

#### Must

- 担当者別実績で作成PR・マージPR・レビューPRの番号一覧を確認できること。
- GitHub Project issueを期間指定で収集し、担当者別のDone件数とEstimate合計を表示できること。
- PRベースとIssueベースを明示的に切り替え可能なメニュー構成を提供すること。
- 不要ファイル scripts/measure-report-prep-time.sh を削除すること。

#### Should

- PR番号詳細表示は、一覧視認性が高く閉じやすいUI（アコーディオンまたはモーダル）で提供すること。
- issue側の集計期間はPR側と同じ入力方式で指定できること。
- データ未取得や空結果時に、利用者が次に行う操作を明示すること。

#### Could

- PR番号一覧から外部リンクで対象PRを直接開けること。
- issueダッシュボードに担当者フィルタやソートを追加すること。

### 4.2 Functional Requirements

- **FR-001**: システムは担当者別実績において、作成したPR、マージしたPR、レビューしたPRの各集合をPR番号単位で保持しなければならない。
- **FR-002**: システムは担当者別実績の各指標セルから、PR番号一覧をアコーディオンまたはモーダルで表示できなければならない。
- **FR-003**: システムはPR番号一覧に表示する要素を、表示中の期間条件および担当者条件に一致するデータのみに限定しなければならない。
- **FR-004**: システムはGitHub Project上のissueを収集し、期間内にDoneになったissueのみを担当者別に集計できなければならない。
- **FR-005**: システムは担当者別にDone issue件数を算出し、一覧表示できなければならない。
- **FR-006**: システムは担当者別にDone issueのEstimate合計を算出し、一覧表示できなければならない。
- **FR-007**: システムはダッシュボード画面にPRビューとIssueビューのメニューを提供し、利用者が任意に切り替えられなければならない。
- **FR-008**: システムは不要ファイル scripts/measure-report-prep-time.sh をリポジトリから削除しなければならない。
- **FR-009**: システムはissueに担当者またはEstimateが未設定のとき、集計ルールに従って欠損扱いを明示しなければならない。
- **FR-010**: システムはPRビューおよびIssueビューのいずれでも、対象データが0件の場合に異常終了せず0件表示を行わなければならない。
- **FR-011**: システムは必要データをローカル環境でシェル実行してJSONとして出力できなければならない。
- **FR-012**: システムはGitHub Pages上で `JSON読込` ボタンを提供し、利用者が選択したローカルJSONファイルから表示を開始できなければならない。

### 4.3 Constitution Alignment

- 既存の収集・集計・表示の責務境界を維持し、最小変更で機能追加する。
- PR番号詳細表示ロジックとIssue集計ロジックを分離し、重複実装を避ける。
- PR番号抽出およびIssue集計ルールに対するユニットテストを追加する。
- GitHub取得境界をモック可能に維持し、オフラインテスト可能性を損なわない。
- 既存Local First方針を維持し、保存済みデータから再表示可能な設計を優先する。

### 4.4 Key Entities

- **PR詳細セット**: 担当者別に「作成」「マージ」「レビュー」の分類とPR番号一覧を保持する集合。
- **ProjectIssueイベント**: issue番号、担当者、Estimate値、Done遷移日時、Project識別子を持つ記録。
- **Issue担当者集計結果**: 担当者ごとのDone件数とEstimate合計を保持する結果データ。
- **ダッシュボードビュー種別**: PRビューとIssueビューの切替状態を表すUI状態。

## ⚠ 5. エッジケース/失敗時の挙動

- 担当者のPR実績が0件の場合: PR番号詳細表示は空一覧を示し、エラーを出さない。
- IssueのDone判定基準が取得できない場合: 対象Project設定不備として通知し、集計結果を未確定として扱う。
- Estimate未設定issueが混在する場合: 件数は集計し、Estimate合計は設定済み分のみ合算した上で未設定件数を併記する。
- 担当者未割り当てissueがDoneになった場合: 未割り当てカテゴリとして集計し、担当者別一覧と区別して表示する。
- PR番号詳細表示の対象件数が多い場合: UIはスクロール可能な一覧で操作不能にならないこと。

## 📚 6. 用語

- 作成PR: 指定期間内に担当者が作成したPR。
- マージPR: 指定期間内にマージ完了したPR。
- レビューPR: 指定期間内に担当者がレビューを行った対象PR。
- Done issue: 指定期間内にGitHub Project上で完了状態へ遷移したissue。
- Estimate: issueに設定される見積値。

## 🎯 7. 成功基準

- **SC-001**: 利用者が担当者別実績からPR番号詳細を3クリック以内で開ける割合が95%以上である。
- **SC-002**: PR番号詳細一覧の内容が集計条件と一致する正答率が100%である。
- **SC-003**: 対象期間のIssue集計で、担当者別Done件数とEstimate合計を100%の担当者で表示できる。
- **SC-004**: 利用者テストで、PRビューとIssueビューを誤操作なく切り替えられる割合が90%以上である。

## 🧭 8. 前提条件

- GitHub ProjectのDone状態はワークフロー列またはステータス値で識別可能である。
- Estimateは数値項目として取り出せる形式で管理されている。
- 期間判定は既存ダッシュボードと同一のタイムゾーンルールを利用する。
- 本機能では新しい認証方式は導入せず、既存のGitHub CLI認証を利用する。
