# Research: PR詳細表示とGitHub Project Issue実績ダッシュボード

## 決定1: PR番号詳細は担当者集計に明示配列を持たせる

- 決定: `createdPrNumbers` / `mergedPrNumbers` / `reviewedPrNumbers` を担当者単位で保持する。
- 根拠:
  - UI側でアコーディオン/モーダルを開いた瞬間に追加再計算せず表示できる。
  - 表示条件（期間・担当者）と集計値の整合性を維持しやすい。
- 検討した代替案:
  - 生PRイベントからフロントで都度フィルタ: 画面責務が増え、整合性検証が難化するため不採用。

## 決定2: Issue実績はDone遷移イベント基準で集計する

- 決定: GitHub Project上でDone状態へ遷移した日時を期間判定に使い、担当者別に件数とEstimate合計を集計する。
- 根拠:
  - 「期間中にDoneになった担当チケット」という要求に直接一致する。
  - 作成日/更新日基準より、実績評価の意味が明確になる。
- 検討した代替案:
  - issueクローズ日で代替: Project上の運用状態とずれる可能性があるため不採用。

## 決定3: Done判定はStatusフィールド名ベース + 設定可能値で扱う

- 決定: Project itemのStatus系フィールド値を参照し、`Done` を既定値として設定可能にする。
- 根拠:
  - Project運用で列名が若干異なるケースに追従しやすい。
  - collector側の設定点で吸収すれば、UI/集計ロジックを固定化できる。
- 検討した代替案:
  - 特定フィールドIDへの固定依存: 環境差分に弱いため不採用。

## 決定4: Estimate欠損は件数に含め、合計値は設定済みのみ加算する

- 決定: Done件数は全件カウントし、Estimate合計は数値設定済みissueのみ合算。未設定件数を併記する。
- 根拠:
  - 実績件数の把握と見積品質の把握を同時に行える。
  - 欠損を0扱いすると実態を過小評価するため。
- 検討した代替案:
  - 欠損を0で加算: 意味的に不正確なため不採用。
  - 欠損をエラー扱い: 運用現実に対して厳しすぎるため不採用。

## 決定5: PR/Issueは同一ページ内タブ切替で導線を統一する

- 決定: 既存Dashboard内に `PR実績` と `Issue実績` のメニュー（タブまたはセグメント）を設ける。
- 根拠:
  - 利用者が同一文脈で比較しやすく、画面遷移コストが低い。
  - 既存画面資産の流用で実装変更量を抑えられる。
- 検討した代替案:
  - 別ページ化: 導線が増え操作が分断されるため不採用。

## 決定6: データ互換はdatasetVersionのminor更新で管理する

- 決定: 既存datasetに後方互換な拡張フィールドを追加し、`datasetVersion` をminor更新する。
- 根拠:
  - 既存PRダッシュボード利用者への影響を最小化できる。
  - 新UIが必要とする情報のみ段階的に追加できる。
- 検討した代替案:
  - 破壊的変更（major更新）: 既存データ再生成コストが高く初期導入に不利なため不採用。

## 決定7: 001-github-activity-dashboardと同一の実行モデルを維持する

- 決定: データ取得はローカル環境でシェルを実行してJSONを生成し、GitHub Pages上では `JSON読込` ボタンからローカルJSONを選択して表示する。
- 根拠:
  - Local First原則に完全一致し、APIキーやサーバ常駐運用を不要にできる。
  - 既存機能（001-github-activity-dashboard）と同じ運用でユーザー教育コストを抑えられる。
- 検討した代替案:
  - GitHub Pagesから直接GitHub APIを叩く: 認証管理が複雑化し、運用コストが増えるため不採用。
  - サーバ経由でJSON生成: 個人利用には過剰であり不採用。
